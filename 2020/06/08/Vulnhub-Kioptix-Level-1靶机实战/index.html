朋友丢了一个虚拟机文件给我，说试试能不能搞进去。下载下来看到是 Kioptix-Level-1 实战靶机，笔者比较菜，平时除了做一些简单的渗透测试之外，没有接触过这类东西，硬着头皮做了一下测试，还请各位师傅轻喷。

# 0x00 实验环境

Kioptix Level 1 靶机：192.168.20.4

kali 攻击机：192.168.20.2.5

# 0x01 实战过程

# 发现目标

打开靶机和 kali 的 vm，在 kali 上使用 netdiscover -r <IP> 来发现靶机。

# 信息收集

得到目标靶机的 IP 后，先做简单的信息收集，上 nmap，发现其开放了 22、80、139 等端口。

浏览器打开该 IP，是一个 apache 的 test page，并没用部署网站，先扫描一下目录。

dirsearch 发现了几个路径，依次打开看下是什么来的。

http://192.168.20.4/usage/ ，内容如下。

http://192.168.20.4/maual 有列目录，点开看到有下面页面。

还有一个 test.php，好像没啥用，用 dirbuster 还扫描出 /mrtg/ 目录等。

到此就没啥头绪了，想到朋友在发我靶机之前，有提起出 nikto，于是在 kali 用 nikto 看下能扫出什么。（突然发现 nikto 还挺好用：））

nikto 的版本和朋友的相同，但是扫出来的结果和朋友发的好像不大一样。

# 漏洞利用

# smb 服务

既然 80 端口没啥利用的，那就转战别的端口吧。靶机是开启了 139smb 服务的，先使用 msf 看下是什么版本的。

smb 版本是 2.2.1a 的，用 searchsploit 查找一下有没有可以利用的漏洞。

有个 RCE，google 一下这个要怎么用。。。

先复制出来用 gcc 编译。

./smb_exp -b 0 <IP> 可以拿到 shell。

# SSL

除了 smb，nikto 好像还扫出可能存在 SSL 的漏洞。靶机的 SSL_version 是 2.8 的。

嗯嗯，有几个缓冲区溢出，先复制出来编译看看。

用法如下：

第一次用， target 参数好像是要输入下面的序号，靶机是红帽，且 apache 版本是 1.3.20。

./ssl 0x6a 192.168.20.4 443 -c 50

诶？Good Bye 是啥意思？google 一下才发现有时候需要打几次，emmm。好吧。

打了几次还是不行，把 - c 参数后的数值改成 100，打进去了，成功 getshell。

可把我激动坏了。

# 0x02 总结

这段时间真的浑浑噩噩的，学啥都学不进去，还真的感谢这位朋友。

这次实战学到很多东西，包括 exploitdb 里面的.c 文件的编译，以及 head 命令查看怎么编译.c 文件。

第一次用 SSL 这类漏洞 getshell，感觉很爽。
